# Harness 工程指南

本文是 [HARNESS-SPEC.zh-CN.md](HARNESS-SPEC.zh-CN.md) 的配套读物。规范用一句话告诉你**该做什么**以及**为什么**。本指南则教你**如何思考** Harness（智能体工作环境框架）的设计，以便应对规范未覆盖的情形。

请先读规范。当你需要理解某条规则背后的理由、看到它在具体场景中的应用，或在决策点需要自行判断时，再来阅读本文。

---

## 1. 为何需要 Harness 工程

没有 Harness 的编程智能体（Agent）就像没有工作台的动力工具：能力再强，若没有稳固台面、夹具、护罩和清晰的工作区，力量反而会碍事。

朴素做法——「给智能体一个任务，放手让它做」——在小型、定义明确的问题上可行，但会以三种可预期的方式崩坏：

**漂移（Drift）。** 在长会话中，智能体会失去连贯性：忘记早先决定、过早收尾（「上下文焦虑」），或产出与一小时前自相矛盾的工作。表面忙碌，却不成体系。

**自我欺骗（Self-delusion）。** 被要求评价自身工作时，智能体往往说「很好」。这不是某个模型特有的缺陷，而是自我评价的结构特性：同一会话里做出的取舍，在被要求复盘时会被合理化——正如人类也难客观评判自己的文章。

**熵增（Entropy）。** 智能体生成的代码会复制既有模式，包括坏的那些。某个文件里的小捷径，几天内就可能变成全库惯例。若不主动纠偏，质量会单调下降。这不是模型能力上限，而是任何「不理解意图、只复制模式」的系统的共性。

Harness 工程同时应对这三点：Harness 提供防止漂移的结构（执行计划、上下文重置）、防止自我欺骗的隔离（独立评估者），以及防止熵增的机械约束（ linter、测试、自动化清理）。这些都不需要模型更聪明，而是需要**环境**设计得更好。

## 2. 三种心智模型

在搭建 Harness 之前，请内化下面三个观念，它们会指导每一次决策。

### 2.1 闭环（The Loop）

> 规范引用：§0.3 公理 3

任何智能体工作流都是一个闭环：智能体行动，环境反馈，智能体再调整。这在任何尺度上都成立——单次修测试的循环、跨多个 sprint 的构建、通宵跑的机群任务。

闭环质量决定产出质量。好的闭环具备：

- **反馈要快。** 智能体能迅速知道上一步是否有效。1 分钟的测试套件让它每次改动后都能自检；20 分钟的套件则迫使它堆一批改动再赌一把——错误会在无人察觉的情况下累积。
- **反馈要诚实。** 信号必须反映现实。linter 对违规视而不见、测试有洞、覆盖率报告排除某些文件——都会造成虚假自信。智能体以为做完了，其实没有。
- **结构性纠偏。** 当同一类失败反复出现，不能只修现象，而要改环境，使问题不再发生：加一条 linter 规则、加一个测试、改工具的接口。每一次修复都让闭环**永久**更聪明。

把闭环看成学习系统。你的任务不是逐个修补输出，而是**改进反馈机制**，让智能体自己修正输出。

### 2.2 脚手架与基本功

> 规范引用：§4.3 Harness 演进

Harness 里有两类组件，混淆它们会导致过度工程或工程不足。

**脚手架（Scaffolding）** 补偿的是模型**尚不能**稳定完成的事。Sprint 拆解存在，是因为当前模型在极长任务上会失焦；上下文重置存在，是因为仅靠压缩并不能根治上下文焦虑；独立评估者存在，是因为模型难以可靠评判自身工作。

这些本质上是临时的。当模型能力提升、能原生承担该任务时，脚手架就变成负担，正确做法是拆掉。

**基本功（Fundamentals）** 则与模型能力无关，始终有价值。linter 维护不变量，因为任何「复制再改」的系统（人或 AI）都自带熵增；类型消灭非法状态，因为边界上的歧义对任何编码者都会造成静默失败；快速测试收紧闭环，因为慢反馈会伤害任何迭代过程。

这些是长期的。即便假设存在「完美模型」，在基本功扎实的环境里仍会做得更好。

实用判别：「一位极优秀的人工程师也会从这条约束中受益吗？」若是，则是基本功；若否，则是脚手架。

### 2.3 智能体失效类型学

智能体以可辨识的方式失败。认出模式，就知道该修 Harness 的哪一块。

| 失效模式 | 你看到的现象 | 根因 | Harness 对策 |
|----------------|-------------|------------|-------------|
| **过早完成** | 工作只做一半就宣称「完成」 | 上下文焦虑；连贯性丧失 | 上下文重置；更完善的 ExecPlan，含明确完成标准 |
| **自信但错误** | 声称功能可用，实际不行 | 自我服务式的评价偏差 | 独立评估者；对真实应用做测试 |
| **模式复制** | 新代码照搬别处的坏模式 | 无强制约束；智能体模仿所见 | linter 规则；黄金原则；针对性重构 |
| **工具静默失败** | 工具报错后智能体仍继续 | 工具返回含糊输出 | poka-yoke（防呆）：重设计工具，使其大声、清晰地失败 |
| **范围膨胀** | 智能体擅自加需求外的功能 | 提示过泛；无验收标准 | 带边界里程碑的 ExecPlan；sprint 契约 |
| **边界腐坏** | API 数据形态错误导致 DB 插入失败 | 系统边缘未类型化 | 在边界解析；端到端类型 |
| **逃避测试** | 写代码却跳过或敷衍测试 | 无覆盖率约束；测试太慢 | 100% 覆盖率目标；快速套件 |
| **决策瘫痪** | 本可自决却反复求助 | 提示过度约束；指令互相矛盾 | 简化 AGENTS.md；消除矛盾 |

出事时，不要只修输出。找出模式、定位根因、加强闭环。

## 3. 入门：Level 1

Level 1 是所有人的起点，也是杠杆最大的层级。搭建得当的 Level 1 Harness 往往让 Level 2 完全不必上场。

**关于顺序。** 下文按叙事展开——从仓库 setup 到第一次跑闭环——**不是**采纳优先级顺序。若你分阶段采纳，规范（§1「采纳优先级」）建议从：仓库知识（§3.1）、快速测试（§3.4 中的测试部分，规范 §1.5）、安全（§3.5）开始——因为它们能闭合反馈环并防止破坏。之后再按需加入代码可读性（§3.2）、工具设计（§3.3）、机械约束（规范 §1.4）与开发环境速度（§3.4 中的环境部分，规范 §1.6），顺序以你最常出现的失效模式为准。

### 3.1 搭建仓库

> 规范引用：§1.1 仓库知识

从三个文件开始：

**`AGENTS.md`**——当作在 onboarding 一位能力强但**零上下文**的外包工程师：会写代码，但对项目一无所知。他们需要知道什么才能上手？

典型项目里一份好的 AGENTS.md：

```markdown
# AGENTS.md

## Project
Invoice processing service. TypeScript + Fastify + PostgreSQL.

## Quick Start
pnpm install && pnpm dev     # starts on port 3000
pnpm test                     # full suite, ~45 seconds

## Architecture
See ARCHITECTURE.md for domain map and layer rules.

## Conventions
- Strict TypeScript; no `any`
- All API endpoints defined in OpenAPI (see openapi.yaml)
- DB types generated by Kysely; never write raw SQL types
- Parse external input at the boundary (see docs/boundaries.md)
- Max file size: 300 lines

## Key Directories
src/domains/         # business logic, one dir per domain
src/infrastructure/  # DB, HTTP, messaging adapters
src/api/             # route handlers (thin; delegate to domains)
tests/               # mirrors src/ structure
docs/                # design docs, decision records
```

注意**没有**写进来的东西：不写长篇决策缘由、不写历史、不写教程——那些放在 `docs/`。AGENTS.md 是速查卡，不是手册。

**`ARCHITECTURE.md`**——叙述系统如何组织（见规范 §1.1）。列出每个顶层目录。说明依赖方向。点出新同事最可能搞错的三件事。

```markdown
# Architecture

## Domain Map
Invoice → LineItem → TaxRule → Currency

## Layer Order (dependency flows downward only)
Types → Config → Repository → Service → Runtime → UI

## Directory Structure
src/domains/invoicing/    # core business logic
src/domains/tax/          # tax computation, isolated from invoicing
src/infrastructure/db/    # Kysely adapters; no business logic here
src/infrastructure/http/  # Fastify plugins and middleware
src/api/                  # route handlers — thin, delegate to services

## Common Mistakes
1. Putting business logic in route handlers (belongs in src/domains/)
2. Importing from src/api/ into src/domains/ (violates layer direction)
3. Writing raw SQL instead of using Kysely typed queries
```

**`docs/core-beliefs.md`**——你的黄金原则：定义代码库「身份」的强硬规则。可视为每次代码审查都要对照的标尺。

```markdown
# Core Beliefs

These are non-negotiable. Every PR, human or agent, is judged against them.

1. **Explicit error handling.** No blanket try/catch wrappers. Every error
   path returns a typed result or throws a domain-specific error.
2. **No business logic in route handlers.** Handlers parse input, call a
   service, and format the response. Nothing else.
3. **Every API endpoint has an integration test.** Unit tests are not
   sufficient for endpoints — the route, middleware, and serialization
   must be exercised together.
4. **Parse at the boundary.** External input is validated into typed
   representations at the edge. Inside the boundary, trust the types.
5. **No orphan code.** If a function isn't called, delete it. If a type
   isn't used, delete it. Dead code misleads agents.
```

### 3.2 让代码库对智能体友好

> 规范引用：§1.2 代码可读性

**采用构建期类型检查。** 这是规范对可读性要求中义务最高的一条：MUST（必须）。若开新项目，优先选自带类型检查的语言（TypeScript、Go、Rust、Kotlin）。若已有动态类型代码库，采用渐进类型：启用严格类型检查器（Python 用 mypy/pyright，JavaScript 用 TypeScript），对新文件与变更文件强制完整类型覆盖，再逐步扩大范围。智能体把类型当作数据形态的首要事实来源——没有类型就会猜，猜测在系统边界会滚雪球成静默失败。

另有两项改动，投入小、收益高：

**拆分大文件。** 实用指引：尽量让单文件低于 300 行。（规范要求「多而小、职责清晰的文件」，但未规定具体数字——300 行是合理阈值，多数模型的工作上下文能完整容纳而不被截断。）1500 行的文件要么被截断（丢掉关键上下文），要么吃掉本该留给任务的预算。不必大改逻辑，只需把内聚片段抽到独立文件，并起清楚名字。

**改掉泛化路径名。** 花 20 分钟把 `utils/helpers.ts` 改成 `invoicing/tax-calculation.ts`，把 `types/index.ts` 改成 `invoicing/types.ts`，智能体在仓库里导航会立刻改善。目录名与文件名是它判断「该去哪找」的首要启发式。

### 3.3 设计智能体工具

> 规范引用：§1.3 工具设计（ACI）

智能体—计算机接口（Agent-Computer Interface，ACI）是智能体调用的工具、命令与脚本集合。即便你没意识到，你其实已经在设计 ACI——仓库里每个 shell 脚本、CLI 封装、构建命令都是其中一部分。

核心原则是 **poka-yoke（防呆）**：让工具难以被误用，而不是指望智能体「自觉用对」。

**实例：**

- 部署脚本若接受 `--env production` 这类旗标，很容易被误用。若改为从与当前分支绑定的配置文件读取目标环境，就是 poka-yoke——智能体无法从功能分支「失手」部署到生产。
- 若数据库迁移工具接受原始 SQL 字符串，容易注入与 schema 错误。若改为由 schema diff 生成类型化迁移文件，就把智能体约束在合法操作上。
- 若文件工具接受相对路径，智能体切换工作目录后就会出问题（而且一定会切）。应要求绝对路径。

**把工具文档当提示来写。** 工具的描述、`--help`、错误信息会直接注入智能体上下文。`--help` 里若清楚展示示例调用、合法输入范围、常见错误，对智能体而言就是自文档化。若只返回晦涩错误码，智能体只能猜。

**I/O 格式要自然。** 智能体生成与解析 markdown、纯文本，通常比 JSON 转义代码块或带 chunk 头的 unified diff 更可靠。设计工具输出时，优先选模型在训练数据里常见的那种「像人话」的文本格式。

### 3.4 速度税

> 规范引用：§1.5 测试、§1.6 开发环境

两项投入往往在首周内就能回本：

**快速测试。** 现在就量一下套件耗时。若超过 60 秒，这是你的头等大事。每多一分钟，智能体在每段会话里能得到的反馈轮次就少一轮。常见优化：并行跑测试、mock 外部服务、单测用内存数据库、缓存第三方 API 响应。

**一键环境。** 写一条脚本，能创建全新、隔离、可工作的开发环境。目标耗时在 2 秒以内。典型实现：`git worktree add`、复制 `.env`、`pnpm install --frozen-lockfile`、启动 dev server。若任一步需要人工（改配置、交互式迁移、在 UI 里点选），就把它自动化，否则你永远难以并行扩展。

### 3.5 动手前的安全

> 规范引用：§1.7 安全基础

在跑第一次智能体闭环之前，先划好安全边界。这很容易被跳过（「以后再补」），但跳过很危险——未沙箱、又持有你凭证的智能体，等于等一个触发条件就会酿成安全事件。

**沙箱化执行环境。** 智能体会执行任意代码——这正是目的。应在容器、虚拟机或隔离的 worktree 中运行，且**不能**触达生产系统。规范将此项定为 MUST，理由充分：除非环境从物理上切断了通路，否则智能体不会区分「测试库」与「生产库」。

**允许列表，而非禁止列表。** 明确列出智能体 MAY（可以）访问的网络主机、shell 命令与文件路径；其余一律默认拒绝。这反直觉，显得「很严」，但智能体会用尽一切可用工具。允许列表意味着误发的 `curl` 打到生产 API 会得到权限错误，而不是改掉线上数据。

**密钥不进仓库。** 智能体会读整个仓库。若 `.env` 或 API key 被提交，就会进入上下文，并可能出现在输出、日志或生成代码里。密钥应通过沙箱运行时的环境变量注入，**不要**放在工作树里的文件中。

**单人开发者的最小配置：**

1. 用 Docker 或同类容器跑智能体，只挂载项目目录。
2. 网络规则：允许 `localhost` 与包仓库镜像；其余阻断。
3. 使用已在 `.gitignore` 中的 `.env`，在容器启动时复制进去——绝不提交。
4. 若项目要连外部服务（数据库、API），只指向本地或测试实例。

大约 30 分钟可搭好，且基本一劳永逸——同一套沙箱配置可服务之后每一次智能体会话。

**设定智能体的默认行为。** 即便有沙箱，智能体仍可能遇到在允许范围内却具破坏性或歧义的操作（例如删测试库表，或需求可作两种理解）。在 Level 1，人类直接监督，智能体默认应**上报并等待**，而非擅自继续。在 AGENTS.md 中写清楚：「遇到破坏性操作或模糊需求时，停止并询问。」（Level 2 会把这形式化为完整上报协议——见 §4.5。）

### 3.6 第一次智能体闭环

基础打好后，闭环本身很简单：

1. 给智能体一个边界清晰的任务。例如：「增加 `GET /invoices/:id`，返回带 line items 的发票，未找到则 404。」不要说「把开票功能做完」。
2. 让它干活。智能体读 AGENTS.md、找相关文件、写代码、跑测试。
3. 观察输出。是否遵守约定？是否跑了测试？测试是否通过？
4. 若出问题，自问：**这是一次性失误，还是模式？** 一次性失误——直接告诉它怎么改。若是模式——要改进闭环：加 linter 规则、澄清 AGENTS.md、改进工具。

久而久之，出错会越来越少，因为每次修复都会**永久**改善闭环。这是 Harness 工程的核心动力学：**你不是在修输出，而是在改进产生输出的系统。**

### 3.7 Level 1 常见误区

**过度提示。** 每个任务写 2000 字提示。若需要这么多说明，问题在环境（缺 AGENTS.md、架构不清、无约定），不在提示。应一次性修好环境，而不是每次任务重讲一遍。

**约束不足。** 约定只写在文档里，不进 linter。智能体读完文档后，随着上下文窗口被占满，会逐渐无视文档约定。而 linter 规则忘不了——过不了构建。把最常被破坏的约定提升为 linter 规则。

**逃避速度税。** 因为「能跑」就容忍慢测试套件。对人来说，一小时测几次可以接受；智能体是每次改动后都测——规范要求 ≤1 分钟（§1.5），因为超过阈值的每一秒都是它少做的一轮反馈。每拖延一周，就少一周的复利。

**逐行审读。** Level 1 时很容易把智能体生成的代码逐行看完。这不具规模，且在 linter、类型、测试扎实的情况下并不必要。重点审关键路径上的变更；其余交给机械检查。

## 4. 扩展：Level 2

### 4.1 何时需要 Level 2

> 规范引用：§2 Level 2——多智能体 + 规划（升级条件）

不要预防性升级。应在观察到具体失效后再升级：

- **智能体反复在完成任务前耗尽上下文**，即便提示已经收得很窄。任务体量本质上超出一个会话能承载的范围。
- **输出质量不稳定**，且智能体无法可靠自我纠正。本该在到你眼前之前就被拦住的 bug，却由你来抓。
- **任务需要前置设计**，而智能体跳过设计直接写代码，导致结构性返工。

若 Level 1 闭环运转正常，只是你心急——请克制升级冲动。Level 2 会引入真实的编排复杂度。

### 4.2 写第一份 ExecPlan

> 规范引用：§2.2 执行计划（ExecPlan）、附录 A.1

ExecPlan 是最难写对的工件，也最能决定 Level 2 成败。常见错误是把它当任务列表。它不是。它是**自包含的操作说明**：陌生人只凭它和当前代码库就能复现你的意图。

试金石：假设删掉所有会话的聊天记录与上下文。一个全新的智能体，**只读** ExecPlan 与当前代码库，能否正确接手并继续？若不能，计划的自包含性还不够。

实用建议：

- **先写 Purpose。** 若用两三句话说不清用户能得到什么，说明功能尚未定义清楚，不宜开做。
- **里程碑按可观察结果定义，而非按代码改动。** 「Milestone 1 之后，执行 `curl localhost:3000/health` 得到 `{"status":"ok"}`」优于「Milestone 1 之后 health 端点已实现」。前者任何智能体都可验证；后者仍留有解读空间。
- **存在不确定性时，加入原型里程碑。** 若不确定某库是否如你所想、或某方案是否可行，先把第一个里程碑做成验证假设的 PoC——比在 Milestone 4 才发现假设错误便宜得多。
- **持续更新。** 计划会随工作推进积累组织知识。Surprises 记录让智能体意外之处——往往对下一会话或下一项目最有价值。

### 4.3 校准评估者

> 规范引用：§2.1 智能体角色、§2.5 质量准则

评估者的好坏取决于校准。开箱即用时常过于宽松：能指出真问题，却又说服自己「问题不大」。

校准闭环：

1. **对你已亲自评过的一份工作跑评估者。**
2. **对比分数。** 评估者与你的判断差在哪？太松？太严？缺了某类维度？
3. **更新评估者提示。** 加入 few-shot，展示你在具体场景下的打分。「该设计在原创性上 3/5，因为只用了默认库样式、无定制」比「对原创性要严格」更有用。
4. **重复**，直到在 3～5 个测试用例上与你的判断对齐。

通常需要 3～5 轮迭代。投入集中在前端——一旦校准稳定，跨次运行都可靠。

校准良好的评估者能抓住真问题。据 Anthropic 实验，评估者反馈示例：

> "Rectangle fill tool only places tiles at drag start/end points instead of filling the region. fillRectangle function exists but isn't triggered properly on mouseUp."

> "PUT /frames/reorder route defined after /{frame\_id} routes. FastAPI matches 'reorder' as a frame\_id integer and returns 422."

这种粒度——点出函数、路由、具体错误——才让评估反馈可执行。含糊的「有些功能不太完美」对生成端没有帮助。

### 4.4 实践中的上下文重置

> 规范引用：§2.4 上下文策略、附录 A.3

理论简单：智能体退化时，清空上下文并重新开始。实践需要判断。

**如何察觉退化：**
- 计划显示进度 40%，智能体却开始收尾（「主要功能已完成」）。
- 质量明显下降——不再跑测试、走捷径、产出与早先决定矛盾。
- 开始重复自己，或重复解释已经做过的事。

**如何执行重置：**
1. 告诉智能体：「我们交接一下。请写一份 handoff 工件总结你的工作。」
2. 智能体把 handoff 写入仓库内文件（例如 `docs/exec-plans/active/handoff-003.md`）。
3. 开启新的智能体会话，指向 ExecPlan 与 handoff。
4. 新智能体读两者，理解当前状态后继续。

**容易出什么问题：**
- 离场智能体写的 handoff 过简，只说「实现了 API」，未写明哪些端点、测了什么、未测什么。接手方重复劳动。**对策**：前几份 handoff 人工过目；若智能体一贯写得太少，调整 handoff 模板或提示。
- 接手方无视 handoff 从头来。**对策**：把 ExecPlan 的 Progress 定为权威状态；接手方看到已勾选里程碑与未勾选后续步骤。

### 4.5 人类上报协议

> 规范引用：§2.6 Level 2 安全

上报协议最难的部分不是定义触发条件，而是**相信智能体真的会停**。

智能体被训练成「乐于助人」，不愿说「我做不到」。若没有明确的停止条件，它会尝试破坏性操作、对模糊需求瞎猜、对失败步骤无限静默重试。

解法是把「停止」做成一等公民行为，而不是失败。在评估者提示与生成端的 AGENTS.md 中，可这样表述：

> "Escalating to a human is the correct action when [trigger conditions]. When escalating, report: (1) what you attempted, (2) what specifically blocked you, (3) your recommended next step. Then stop and wait."

用故意模糊的任务做测试，验证它会选择上报而不是瞎猜。

## 5. 做大：Level 3

### 5.1 何时需要 Level 3

> 规范引用：§3 Level 3——集群与机群（升级条件）

Level 3 不是「Level 2 加强版」，而是质变：从顺序流水线到并行蜂群。信号包括：

- 待办 backlog 持续深于流水线吞吐，功能在排队等生成/评估循环。
- 任务足够独立，多个智能体可同时处理不同项而互不踩脚。
- 已在 Level 2 基础设施（ExecPlan、评估者、上下文重置）上投入且运行稳定。

若 Level 2 流水线吃不饱——瓶颈在构思或规格，而非执行——你不需要 Level 3，需要的是更好的规划。

### 5.2 合并队列问题

> 规范引用：§3.3 合并队列

一旦有两个智能体并行工作，就出现合并队列问题。智能体 A 完成并合并。智能体 B 也完成了，但基线已变。B 的改动可能与 A 冲突——不仅是文本层面（git 能发现），还有语义层面（同一函数为不同目的被修改）。

朴素做法——让智能体自己 rebase、merge——会退化成 Yegge 所说的「猴子拿刀互砍」：智能体花在解决冲突上的时间超过写功能。

解决方式是专用合并智能体：

1. 从队列依次取 PR。
2. 对当前 HEAD 做合并。
3. 若有冲突，阅读双方分支，理解各自意图，产出正确合并。
4. 对合并结果跑完整测试套件。
5. 测试通过则合并；否则把 PR 退回并附上具体失败信息。

这在设计上是**顺序**的。把合并本身并行化是假优化——协调成本超过省下的时间。

### 5.3 让工作可持久

> 规范引用：§3.1 工作流持久性

在 Level 3，智能体会崩溃、过期、被替换。只有进度能扛过这些扰动，系统才成立。

关键认识：**会话不是工作单元——ExecPlan 中的步骤才是。** 智能体认领一步、执行、标记完成。若中途挂掉，该步保持未认领。下一个智能体捡起它，检查当前状态（改动是否只应用了一半？测试是否通过？），再完成或重启该步骤。

因此 ExecPlan 的步骤必须**幂等**。智能体应能从头重跑任一步，而不留下重复或损坏。

步骤认领必须**原子化**——同一时刻只有一个智能体处理某一步。否则两个智能体可能同时认领同一步，产生冲突变更。

实用模式：每个智能体启动时读被指派的 ExecPlan，**原子地**认领第一个未认领步骤并开始；完成后标记完成、提交进度更新，再进下一步。若即将耗尽上下文，提交 handoff 并干净退出。

## 6. 维护 Harness

### 6.1 实践中的熵增

> 规范引用：§4.1 熵管理

你最先会感到「这代码库越来越乱」，随后看到具体症状：重复工具函数、错误处理风格不一、命名惯例混杂。

**在 Level 1**，熵管理是人的责任。在日常审查智能体输出时，留意模式漂移——智能体把某文件的捷径复制到五个新文件。当某类偏离反复出现，不要仅在审查中修补：将修复提升为 linter 规则（参见 §3.7 关于约束不足）。每加一条规则，就永久消灭一类熵增来源。

**在 Level 2 及以上**，将原先手动的检查自动化。后台智能体夜间扫描，对照 `docs/core-beliefs.md` 中的黄金原则查找偏离。它提交小而专的 refactor PR——每个只修正一类模式偏离。这类 PR 往往一分钟内即可完成审查，且常可自动合并。

两种情况的关键都是**频率**。每周一次、堆出 500 行的大 PR，难审、难合。每晚五个 20 行的小 PR，几乎无感。技术债管理与复利同理：**小额、高频**优于偶发大额。

### 6.2 何时简化

> 规范引用：§4.3 Harness 演进

模型升级后，安排一次「简化 sprint」：系统性地验证每项脚手架是否仍不可或缺。

方法：

1. 列出所有脚手架组件（sprint 拆解、上下文重置、独立评估者等）。
2. 逐项关闭，跑代表性任务。
3. 对比有无该组件时的输出质量。
4. 若质量相当，永久移除该组件。

对结果要诚实。容易留恋「感觉有用」却无法量化提升的组件。每个多余组件都会增加延迟、成本与编排表面积。

典型结果：有些明显仍需要，有些明显过时，还有一些灰色地带——难任务有用、简单任务成负担。对灰色地带可考虑改为可选，而非默认开启。

### 6.3 演进 Harness

Harness 是活的系统，不是一次性 setup。预期每隔几周就会有实质性改动，驱动来自三方面：

**模型变化。** 新模型可能让某些脚手架过时，或带来新能力。跑简化 sprint（§6.2）。

**失效模式。** 若某类失败重复出现，追溯到 Harness 缺口。纪律是：不只修输出，而要修**产生输出的闭环**。久而久之，Harness 会高度贴合你的项目。

**规模变化。** 团队变大、加新服务、进新领域——都可能改变最优 Harness 配置。为 10 个文件的微服务设计的 Harness，不经调整很难直接服务 500 个文件的 monorepo。

健康 Harness 的标志：人类介入频率随时间下降，吞吐与质量上升。若你越往后越要救场，说明 Harness 里积压了未闭合的反馈环。

## 7. 背景：我们如何走到这里

本节是脉络，不是规定。它说明规范中实践的思想渊源。

### 7.1 趋同的发现

2024 末至 2026 初，多个团队独立得出了惊人相似的结论：什么让智能体有效。这不是事先串通——他们在解同一道题，因而触达同一套「物理规律」。

**OpenAI 团队**在五个月内做了一个几乎零手写代码的内部产品。关键发现：**仓库本身**——而非提示、也非模型——是智能体效能的首要杠杆。他们把代码库当作单一事实来源，把上下文推进版本化工件，并用机械手段强制架构不变量。「AGENTS.md 作目录」、渐进披露、文档园艺等做法，都来自「智能体可读性是瓶颈」这一认识。

**Anthropic 团队**探索了两个方向：前端设计（主观质量）与全栈应用（可验证正确性）。核心发现：把生成与评估分离——他们自述受 GAN 启发——在实验中带来最大的可测质量提升。他们还把上下文焦虑列为首要问题，并报告对表现该问题的模型，上下文重置优于压缩。其简化原则——模型长大后就拆掉 Harness 组件——让 Harness 长期保持精干。

**Geoffrey Huntley** 阐述了闭环原语与「先单体后复杂」原则。其「Ralph Wiggum Loop」——智能体与环境之间持续迭代——成为广泛采用的心智模型。核心论点：在单智能体闭环**明确证明**解决不了问题之前，不要上多智能体复杂度。

**Steve Krenzel（Logic.inc）** 表明传统工程最佳实践——100% 测试覆盖、快速套件、类型化边界、语义化命名——在智能体语境下价值陡增。对人可选的东西，对智能体往往**必需**，因为智能体会放大代码库里已有的一切，无论好坏。

**Steve Yegge** 用 Gas Town 把边界推远，证明在正确基础设施下，大规模智能体编排（30+ 并发）可行：git 中的持久工作态、专用合并队列、巡逻智能体监控 worker 健康。其「非确定性幂等」概念——工作流从构造上扛住智能体崩溃——是 Level 3 持久性的基础。

### 7.2 仍未定论之处

Harness 工程尚年轻。若干问题仍开放：

- **架构连贯性如何随年演进？** 当前实验跨度以月计。尚无人知道「五岁、全智能体生成」的代码库长什么样。
- **人类判断在哪最有杠杆？** 「智能体能决定」与「人必须决定」的边界随代际模型移动。最优分工是移动靶。
- **Harness 如何适配多模态智能体？** 当智能体能看截图、听音频、推理视频，评估面会扩张。当前 Harness 仍以文本为主。
- **成本模型应如何建立？** 按每小时约 $10–12 计，Level 3 日成本可达数千美元。产出质量能否证明其物有所值？在何种条件下成立？行业尚缺稳健基准。

这些是研究问题，不是拦路虎。用今天可行的方式搭建；答案浮现时再演进。
